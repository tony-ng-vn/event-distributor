Fix event feed dropping events after 24 hours

At a glance

Events were saved in Postgres but disappeared from the app about a day after their start time. The feed API filtered out anything older than 24 hours. This commit removes that filter and sorts upcoming events first, then recent past events.

Background

Deep background (skippable)

The app stores shared events in InsForge Postgres. Ingest writes rows via POST; the feed reads them via GET /api/events → listFeedEvents().

What mattered before

fetchUpcomingEventRows() used .gte('start_at', now - 24h). Once an event's start time was more than a day ago, every feed reload hid it — even though the row still existed.

Intuition

Goal: If someone adds an event to the shared feed, it should stay visible until an admin deletes it — not vanish because a server-side date window expired.

Postgres row→API query→Feed UI

Persistence was fine; the read filter made it look like data loss. Remove the cutoff, then sort so upcoming events appear first (nearest first) and past events follow (most recent first).

Code walkthrough

Step 1 — Remove the 24-hour cutoff

fetchAllEventRows() selects all events without .gte('start_at', cutoff), then applies sortFeedEventRows.

const { data } = await db.database
  .from('events')
  .select(eventSelect)
  .order('start_at', { ascending: true });
const events = sortFeedEventRows((data ?? []) as InsforgeEventRow[]);
Step 2 — Sort upcoming first

sortFeedEventRows splits on start_at >= now: upcoming events nearest-first, then past events most-recent-first.
