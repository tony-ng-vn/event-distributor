At a glance

Research + PRD for filtering the shared feed by event type. Keyword rules fail on playful SF Luma titles; recommend async LLM classification into a small closed taxonomy, then type filter pills.

Background

Deep background (skippable)

Event Radar stores Luma-scraped events (title, description, location, host, is_online) with no category field. Feed filters today are response-state only (all/pending/accepted).

What mattered before

This branch adds docs under docs/prd/ plus a read-only dump script. Production InsForge could not be queried in the cloud agent (missing API key), so research used a public Luma SF discover proxy corpus.

Intuition

Goal: Give humans a clear, implementable plan: taxonomy, classifier contract, and filter UX — without shipping app code yet.

Ingest→async classify→primary_type→feed pills

Insert first so calendar sync never waits on a model. Untyped rows stay under All; Other means we tried and could not type it.

Code walkthrough

Step 1 — Research corpus

Proxy SF titles show multi-intent events (e.g. paddle + AI branding). Keyword maps mis-label the interesting middle.

Step 2 — PRD taxonomy

Closed set: social, builders, talks, sports, arts, other. LLM at classify-time with soft fallback; admin override.

Step 3 — Contract tightening

type_source enum, async post-ingest, Other excludes untyped — so a future agent can implement without guessing.
