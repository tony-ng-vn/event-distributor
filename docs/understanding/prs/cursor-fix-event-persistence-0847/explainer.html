Fix event feed persistence

At a glance

Events stayed in Postgres but vanished from the UI after ~24 hours because the feed API filtered on start_at. This PR loads all events and sorts upcoming first.

Background

Deep background (skippable)

The app stores shared events in InsForge Postgres. Ingest writes rows; the feed reads them via GET /api/events.

What mattered before

fetchUpcomingEventRows() used .gte('start_at', now - 24h), hiding older rows from every client reload.

Intuition

Goal: Keep added events visible until an admin deletes them, while still surfacing upcoming events first.

Postgres row→API query→Feed UI

Persistence was fine; the read filter made it look like data loss. Remove the cutoff and sort sensibly.

Code walkthrough

Step 1 — Remove 24h cutoff

fetchAllEventRows() selects all events without .gte('start_at', cutoff).

const { data } = await db.database
  .from('events')
  .select(eventSelect)
  .order('start_at', { ascending: true });
Step 2 — Sort upcoming first

sortFeedEventRows puts future events at the top (nearest first), then past events (most recent first).

Step 3 — Regression test

Integration test inserts an event from 3 days ago and asserts listFeedEvents still returns it.
